# AI 原生打磨 — 验收标准

## 原则
> 能用 AI 动态生成的，绝不用硬编码规则。
> 每个模块上线前必须回答：这个逻辑是否「千人千面」？

---

## P0: AI 情绪安抚（替换 `_soothe` 关键词匹配）

### 当前问题
`_soothe()` 用 6 组硬编码关键词（焦虑/困惑/悲伤/愤怒/心碎）做字符串匹配，
漏掉大量真实情绪表达（如「好丧」「心态崩了」「emo了」「整个人都不好了」）。

### 验收标准
- [ ] P0.1 用现有的 `MoodDetector`（DeepSeek Flash）替换关键词匹配
- [ ] P0.2 情绪安抚文本由 AI 动态生成（不硬编码 5 种回复模板）
- [ ] P0.3 AI 生成的安抚必须：匹配用户原话 + 1-2 句 + 有温度
- [ ] P0.4 安抚生成速度 < 2 秒
- [ ] P0.5 无情绪时不生成空安抚（AI 判断是否需要安抚）
- [ ] P0.6 测试 20 种真实情绪表达 → 正确识别率 > 85%

### 验收测试
```python
test_cases = [
    ("好丧啊最近", "需要安抚"),
    ("哈哈哈今天太开心了", "不需要安抚"),
    ("心态崩了", "需要安抚"),
    ("帮我看看八字 1990年5月", "不需要安抚"),
    ("emo了 什么都不想做", "需要安抚"),
    ("老板太恶心了想辞职", "需要安抚"),
]
```

## P1: AI 意图识别（替换 `_detect_intent` 关键词匹配）

### 当前问题
`INTENT_KEYWORDS` 用 10 组硬编码关键词做意图路由，用户说「帮我看下命」就不会被识别（因为不包含「八字」两个字）。

### 验收标准
- [ ] P1.1 用 LLM（Flash）替换关键词匹配做意图分类
- [ ] P1.2 支持 11 种意图 + 「自由对话」共 12 种分类
- [ ] P1.3 用户说「看下命」「算一下」「最近运气咋样」→ 正确路由到八字
- [ ] P1.4 用户说「梦见xxx」→ 正确路由到解梦（无论是否含「解梦」二字）
- [ ] P1.5 意图识别速度 < 1 秒
- [ ] P1.6 测试 30 种口语化表达 → 正确率 > 90%
- [ ] P1.7 保留关键词作为快速路径（含出生日期 → 直接八字，不调 AI）

### 验收测试
```python
test_cases = [
    ("帮我看下命", "bazi"),
    ("算一下运势", "bazi"),
    ("昨晚梦见掉牙齿了", "dream"),
    ("这房子风水怎么样", "fengshui"),
    ("给我起个卦", "liuyao"),
    ("他跟我配不配", "hehun"),
    ("今天适合搬家吗", "zeri"),
    ("帮我看看我的名字好不好", "xingming"),
    ("最近运气咋样", "bazi"),
    ("我心里好难受", "free_chat"),  # 情绪聊天，不路由
]
```

## P2: AI 动态情绪开场（替换固定模板）

### 当前问题
解梦、八字、风水等各模块的「开场白」都是硬编码模板，读起来像机器人。

### 验收标准
- [ ] P2.1 解梦的情绪开场由 AI 动态生成（不再硬编码 3 种）
- [ ] P2.2 八字秒回 (`_gen_instant_reply`) 由 AI 动态生成（不再 if-else 拼凑）
- [ ] P2.3 开场白必须包含：对用户处境的呼应 + 对命理结果的个性化解读
- [ ] P2.4 同一用户两次同样输入 → 两次不同开场白（验证非模板）
- [ ] P2.5 各开场白生成速度 < 3 秒

---

## P3: 移除禁词表 + 减少硬编码模板

### 当前问题
`BANNED_WORDS` 用字符串替换处理，但系统 Prompt 已经禁止了这些词。
错误/引导消息有大量硬编码模板。

### 验收标准
- [ ] P3.1 移除 `BANNED_WORDS` 字符串替换逻辑
- [ ] P3.2 系统 Prompt 强化禁止词 → 验证 20 次对话中不出现「小友」「老夫」
- [ ] P3.3 信息收集引导语（「请提供出生信息」等）由 AI 根据上下文动态生成
- [ ] P3.4 handler.py 中硬编码字符串从 ~40 处减少到 < 15 处

---

## P4: AI 驱动的对话流

### 当前问题
整个对话流程是 rigid 的：检测意图 → 路由处理器 → 返回结果。
用户说「焦虑跳槽」时，系统无法灵活地在「情绪聊天」和「命理分析」之间切换。

### 验收标准
- [ ] P4.1 AI 判断当前对话应该「安抚情绪」「收集信息」还是「命理分析」
- [ ] P4.2 用户表达情绪 + 命理问题时，AI 能自然切换（先安抚再分析）
- [ ] P4.3 用户连续问多个问题时，AI 能保持上下文连贯
- [ ] P4.4 对话流决策在 1 个 LLM 调用内完成（不额外增加延迟）

---

## 执行规则

1. 每个 P 级独立验收
2. 每完成一个 P 级：本地验证 → 部署 → 跑 8 场景测试 → 确保不回归
3. 硬编码新代码 < 5 条硬编码字符串
4. 每个 P 完成后用 design-critique 审查

---

## 优先级排序

| 优先级 | 模块 | 预计改动 | 影响 |
|:--:|------|------|------|
| P0 | AI 情绪安抚 | handler.py ~50行 → ~30行 | 情绪体验飞跃 |
| P1 | AI 意图识别 | handler.py ~20行 → ~30行（加 AI 调用） | 路由准确率飞跃 |
| P2 | AI 动态开场 | handler.py ~100行 → ~40行 | 个性化体验 |
| P3 | 移除禁词+模板 | handler.py 净减少 ~30 行 | 代码质量 |
| P4 | AI 对话流 | handler.py 架构调整 | 体验天花板提升 |
