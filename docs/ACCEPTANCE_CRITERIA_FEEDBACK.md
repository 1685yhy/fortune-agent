# 反馈校准闭环 — 验收标准

## 原则
> 系统必须从每次用户交互中学习，越用越聪明。
> 用户偏好应该是「学出来的」，不是「问出来的」。

---

## F1: 用户偏好学习引擎

### 验收标准
- [ ] F1.1 每次用户给 👍/👎 → 更新用户偏好向量
- [ ] F1.2 偏好向量包含：风格偏好(sassy/analyst/gentle)、话题偏好(财运/感情/事业/健康/成长)、长度偏好(简短/详细)
- [ ] F1.3 学习算法：指数加权移动平均（新反馈权重 > 旧反馈）
- [ ] F1.4 至少 3 次反馈后才开始自动调整（避免小样本偏差）
- [ ] F1.5 偏好数据持久化到 SQLite

### 验收测试
```python
# 用户连续 5 次在毒舌模式给 👍 → 系统应自动偏好毒舌
for i in range(5):
    record_feedback(user_id, personality="sassy", rating="positive")
prefs = get_preferences(user_id)
assert prefs["style"]["sassy"] > prefs["style"]["gentle"]
```

## F2: 回复自动适配

### 验收标准
- [ ] F2.1 系统 Prompt 注入用户偏好摘要（不改变回复结构，只微调风格）
- [ ] F2.2 偏好「简短」的用户 → LLM max_tokens 自动降低
- [ ] F2.3 偏好「财运」话题的用户 → 分析中自动侧重财运维度
- [ ] F2.4 偏好数据不可用时 → 优雅降级，不影响正常回复
- [ ] F2.5 适配不增加额外延迟（偏好注入在 Prompt 级别完成，无额外 API 调用）

### 验收测试
```python
user_with_prefs = setup_user(prefers_short=True, prefers_topic="财运")
response = chat("帮我看看运势", user_with_prefs)
assert len(response) < 300  # 简短偏好 → 回复应较短
assert "财" in response      # 财运偏好 → 应包含财运分析
```

## F3: 反馈提示

### 验收标准
- [ ] F3.1 每次命理分析（八字/紫微/风水/解梦等）回复末尾附带自然的 👍👎 提示
- [ ] F3.2 提示风格匹配当前人格模式（毒舌/分析/温柔各有不同的提示语）
- [ ] F3.3 用户发送 👍 或 👎 → 被正确识别并记录
- [ ] F3.4 连续对话中不重复提示（同一会话最多提示一次）
- [ ] F3.5 自由聊天（非命理分析）不提示反馈

### 验收测试
```python
response = bazi_analysis("1990年5月20日 北京 男")
assert "赞" in response or "踩" in response or "有帮助" in response
# 确认反馈提示存在且自然

# 自由聊天不应有反馈提示
response = free_chat("你好")
assert "有帮助吗" not in response
```

## F4: 准确率仪表盘

### 验收标准
- [ ] F4.1 GET /api/user/{id}/accuracy 返回完整仪表盘数据
- [ ] F4.2 包含：总反馈数、准确率%、按话题分类准确率、最近趋势
- [ ] F4.3 按话题分类：财运准确率、感情准确率、事业准确率等
- [ ] F4.4 趋势数据：最近 4 周的准确率变化
- [ ] F4.5 新用户（无反馈数据）返回友好提示而非空数据

---

## F5: 周报个性化

### 验收标准
- [ ] F5.1 周报生成时读取用户偏好数据
- [ ] F5.2 偏好「财运」→ 周报侧重财运分析
- [ ] F5.3 周报内容包含上周准确率数据（如有反馈）
- [ ] F5.4 不同用户同一周 → 内容侧重点不同（验证千人千面）

---

## 执行规则

1. F1-F5 顺序执行，每个 F 级独立验收
2. 每完成一级：单元测试 → 部署 → 跑 8 场景回归 → 确保不破
3. 新代码硬编码字符串 < 5 条
4. 偏好学习算法简单透明（不引入深度学习依赖）
