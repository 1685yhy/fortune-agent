# 第二轮打磨验收标准 — 基于用户调研

## 原则
> 每个功能上线前必须通过验收。每个验收项对应一个具体的用户需求。

---

## S-A: 定期运势追踪（周报系统）

### 用户需求
调研发现 #1 用户渴望：将一次性报告变成持续更新的订阅式服务。

### 验收标准
- [ ] S-A.1 周报基于用户真实八字生成，非模板套话
- [ ] S-A.2 周报包含：本周基调 + 注意事项 + 幸运日/色/方向 + 赋能语
- [ ] S-A.3 同一用户连续2周生成 → 内容不重复（验证非模板）
- [ ] S-A.4 不同用户同一周 → 内容不同（验证千人千面）
- [ ] S-A.5 周报生成速度 < 10 秒
- [ ] S-A.6 每周一早 8 点自动触发（cron 验证）

### 验收测试
```python
# 1. 生成2个用户的本周报告 → 内容不同
# 2. 同一用户连续2周 → 内容不同
# 3. 检查报告包含4个必需部分
# 4. 检查 cron 任务已配置
```

## S-B: 反馈校准闭环

### 用户需求
用户期望对预测结果进行「感知准确度」评分，形成反馈→校准的正循环。

### 验收标准
- [ ] S-B.1 每次分析回复末尾附带 👍👎 反馈提示
- [ ] S-B.2 用户发送 👍 → 系统记录正面反馈
- [ ] S-B.3 用户发送 👎 → 系统记录负面反馈
- [ ] S-B.4 GET /api/user/{id}/accuracy 返回百分比
- [ ] S-B.5 周报中包含"上周预测准确率"数据
- [ ] S-B.6 反馈不影响正常对话流程（用户发了👍照样可以继续聊）

### 验收测试
```python
# 1. 完成一次分析 → 检查回复末尾有反馈提示
# 2. 模拟发送 👍 → 检查数据库 feedback 字段更新
# 3. 查询准确率 → 检查返回百分比
```

---

## S-C: 数据扩充

### 用户需求
更全面、更准确的解读需要更丰富的古籍和案例数据。

### 验收标准
- [ ] S-C.1 至少新增 3 个数据源
- [ ] S-C.2 新增数据包含：八字案例 + 解梦条目 + 风水知识
- [ ] S-C.3 RAG 文档数从 16,560 增长到 > 18,000
- [ ] S-C.4 新增数据可被 RAG 检索到
- [ ] S-C.5 新增数据来源标注清晰

### 验收测试
```python
# 1. 搜索"梦见蛇"→ 检查返回结果数增加
# 2. 搜索"七杀格 案例" → 检查有案例结果
# 3. 检查 RAG 文档总数 > 18000
```

---

## S-D: 周报推送集成

### 验收标准
- [ ] S-D.1 周报可通过 API 手动触发：POST /api/push-weekly
- [ ] S-D.2 周报内容包含用户昵称（如有）和本周日期范围
- [ ] S-D.3 同一用户不会在同一周收到重复推送
- [ ] S-D.4 推送失败不阻塞系统正常运行

---

## 审查流程

每个 Sprint 完成后：
1. 跑验收测试
2. 用 design-critique 审查代码质量
3. 用 quality-checklist 审查整体品质
4. 修复发现的问题
5. 部署 + GitHub
